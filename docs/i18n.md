# 다국어(i18n) 관리 가이드

이 문서는 Summit 프로젝트의 번역 시스템 운영 정책과 용어 관리 방법을 설명합니다.

## 1. 번역 정책 (AI 기반 자동화)

2026년 4월부터 본 프로젝트는 **"한-영 수동 관리 + AI 자동 번역"** 정책을 따릅니다.

- **기본 언어**: 한국어(ko)와 영어(en)는 수동으로 관리하며 모든 번역의 기준이 됩니다.
- **자동 번역 언어**: 중국어(zh), 일본어(ja), 스페인어(es)는 Gemini 1.5 Flash 모델을 사용하여 한/영 원문을 기반으로 자동 생성됩니다.
- **일관성 유지**: DB 기반의 **용어집(Glossary)**을 사용하여 고유 명사 및 등산 전문 용어가 모든 언어에서 일관되게 번역되도록 관리합니다.

## 2. 관리 시스템 구조

### A. 동적 데이터 (DB JSONB)
산(mountains), 경로(routes), 경유지(waypoints), 구간(segments) 등의 동적 데이터는 JSONB 구조를 사용합니다.

- **데이터 구조**:
  ```json
  {
    "ko": "한글 원문",
    "en": "English Source",
    "zh": "AI 자동 생성",
    "ja": "AI 자동 생성",
    "es": "AI 자동 생성"
  }
  ```
- **변경 감지 (`translation_meta`)**: `ko`와 `en` 텍스트의 해시값을 저장하여, 원문이 수정될 경우에만 AI 번역을 다시 실행합니다.

### B. 정적 UI 문자열 (i18n_glossary)
버튼 라벨, 메뉴 이름 등 앱 전반에 사용되는 정적 문구는 Supabase의 `i18n_glossary` 테이블에서 통합 관리합니다.

- **관리 방법**: `category`가 `ui`인 항목들이 앱의 UI 문구로 사용됩니다.
- **코드 동기화**: DB의 UI 문구는 번역 스크립트를 통해 `src/lib/ui_strings_generated.json` 파일로 자동 추출되어 앱에 반영됩니다.

## 3. 용어집(Glossary) 활용

특수 용어는 `i18n_glossary` 테이블에서 `category: 'glossary'`로 관리됩니다. AI는 번역 시 이 테이블을 최우선으로 참조합니다.

- **예시**:
  - `계곡` -> 지형 안내 시 `Valley`, 물을 가리킬 시 `Stream`으로 구분하여 관리.
  - `북한산우이역` -> `Bukhansan Ui Stn` 등 표준화된 명칭 사용.

## 4. 콘텐츠 업데이트 방법

### UI 문구 (버튼, 라벨 등) 추가/수정
1. Supabase **`i18n_glossary`** 테이블에 접속합니다.
2. 새로운 문구의 경우 `key`와 한글(`ko`), 영어(`en`) 내용을 입력합니다. (`category`는 `ui`로 설정)
3. 터미널에서 동기화 스크립트를 실행합니다 (아래 참조).

### 동적 데이터 (산, 경로 설명 등) 수정
1. 해당 테이블의 컬럼에서 `ko` 또는 `en` 필드만 수정합니다.
2. 번역 스크립트를 실행하면 AI가 나머지 언어를 즉시 갱신합니다.

## 5. 번역 및 동기화 스크립트 실행

수동으로 번역을 실행하고 UI 문구를 코드에 동기화하려면 다음 명령어를 실행합니다.

```bash
npx tsx scripts/translate.ts
```

> [!IMPORTANT]
> - 실행 전 `.env.local`에 `GEMINI_API_KEY`가 설정되어 있어야 합니다.
> - 스크립트 실행 후 `src/lib/ui_strings_generated.json` 파일이 변경되었다면 Git에 커밋하여 배포해야 합니다.

## 6. 수동 번역 덮어쓰기
AI가 번역한 내용이 마음에 들지 않을 경우, DB에서 직접 `zh`, `ja`, `es` 컬럼을 수정할 수 있습니다. 한글이나 영어 원문이 바뀌기 전까지 AI는 사용자가 수동으로 수정한 내용을 덮어쓰지 않습니다.
