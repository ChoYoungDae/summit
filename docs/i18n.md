# Internationalization (i18n) Guide

This document outlines the translation system and terminology standards used in the Summit project.

## 1. Translation Systems

The project uses two primary methods for internationalization:

### A. Database-backed Multilingual Fields (JSONB)
Dynamic content stored in the database (e.g., mountain names, route descriptions, direction guides) uses a JSONB structure to support multiple locales.

**Structure:**
```json
{
  "en": "Text in English",
  "ko": "한국어 텍스트",
  "zh": "中文文本",
  "ja": "日本語テキスト",
  "es": "Texto en español"
}
```

**Usage:**
Use the `t(text, locale)` function from `src/lib/i18n.ts` to resolve the correct string. It automatically falls back to English if the requested locale is missing.

### B. UI String Dictionary
Static UI labels and messages are managed in `src/lib/i18n.ts` within the `UI_STRINGS` object.

**Usage:**
Use the `tUI(key, locale)` function to retrieve a translation for a specific UI element.

## 2. Terminology Standards

To ensure consistency and safety, specific terms are standardized as follows:

### Safety Return (안전 귀가)
The "Safety Return" feature calculates the latest time a hiker should start their journey to return before sunset.

| Key | English | Korean (Standard) | Rationale |
| :--- | :--- | :--- | :--- |
| `lastSafeStart` | Last safe start | **안전 산행 마감 시간** | Represents the *deadline* for a safe start. |
| `routesCount` | Routes | **경로** | Used in chips (e.g., "3 보유 경로"). |
| `valueSafetyReturnTitle` | Safety Return | 안전 귀가 가이드 | - |

## 3. Supported Locales
- `en`: English (Default / Fallback)
- `ko`: Korean
- `zh`: Chinese
- `ja`: Japanese
- `es`: Spanish

## 4. How to Add New Translations
1. **UI Strings**: Add the key to both `en` and `ko` (and other locales if available) in `src/lib/i18n.ts`.
2. **Database Fields**: Ensure the JSONB object in the database includes the necessary locale keys.
